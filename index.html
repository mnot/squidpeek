Key

Each line in the results indicates the service statistics for one service URL. Most graphics can be 'moused over' to reveal more detailed statistics for an individual entry.

accesses

This column shows how many acccesses that the URL received during the sample period. It does not include ICP or other inter-cache traffic, nor does it include 'async' traffic caused by stale-while-revalidate.

query diversity

This column shows how many different query arguments were seen for this URL to the left, and a graph of how popular they were to the right.

For example, if a URL http://example.com/foo has 1000 accesses, and a query diversity of 250, it means that 250 different combinations of queries to it were seen. For the purposes of calcuating these metrics, query strings and path parameters are both considered query arguments, so that both http://example.com/foo?bar and http://example.com/foo;baz=bat would be collpased into the URL above (contributing to the 250 figure).

The graph next to it shows how accesses to the queries are distributed; If there is a high peak on the left and a short tail , it means that most accesses went to a few query terms, while if there is a low peak and a long tail , it means that the queries were distributed over a larger set of request-URIs. Mousing over the graph will show how much of the total accesses went to the most popular query, as a percentage.

In general, a large query diversity means that traffic to a particular service is more difficult to cache; as the query diversity number approaches the number of accesses, there is less for the cache to exploit, and the hit rate will go down. However, if a reasonble amount of traffic goes to the most popular query terms, it is still possible to achieve a decent hit rate.

hits

This column shows the percentage of hits for this URL on the left, and a graph representing their distribution on the right.

In these results, a hit is a response that is served without needing to contact another server. Hits are very fast and do not cause any load on the systems behind the cache; high hit percentages are desirable.

negative hits (red)
Cached errors. Certain error response status codes are cached, so that the server isn't overwhelmed. A large number indicates that a lot of errrors are coming from your service.
disk hits (grey)
Hits served from disk. Fast, but slower than from memory. A large number indicates that this service isn't 'hot' enough to be memory cached consistently.
memory hits (green)
Hits served from memory. Fastest possible service.
stale hits (yellow)
Hits served stale, usually because of mechanisms like stale-while-revalidate. A small number indicates that these mechanisms are working correctly. A large number indicates that the service is taking a long time to revalidate, and/or has a small freshness lifetime.

misses

This column shows the percentage of misses for this URL on the left, and a graph representing their distribution on the right.

In these results, a miss is a response that requires some communication with an upstream server -- usually the origin server, or another cache. Misses are slower and cause load on other systems, and so are less desirable. However, they are unavoidable (the cache has to fill somehow), and in some situations are desirable because the back-end system has to be contacted.

client no-cache (yellow)
Client asked for a fresh copy, with Pragma and/or Cache-Control request headers. A large number indicates misconfigured and/or aggressive clients.
no validator (grey)
No validator present in the cached response, so the cache was forced to fetch a new copy. Validators like Last-Modified and ETag on responses allow caches to ask of a response has changed, rather than getting a whole new one. This can save bandwidth if responses are large, and in some situations can avoid server-side computation. A large number indicates that this feature is not being used.
validate unsuccessful (yellow)
Validator present, but validation unsuccessful. A validator was present in the cached response, but the server sent a new copy when contacted. A large number indicates that the server doesn't support validation (even though it is sending response headers that can be used as validators), or that the responses are changing quickly on the server.
validate successful (green)
Validator present, validation successful. A validator was present in the cached response, and the server indicated that the same response can be used; a new response was not fetched. A large number indicates that the server is sending validators and responding to validating requests.

miss msec

This column shows the median miss time on the left, and a one-second wide histogram of miss times on the right, both in millisecond units. This indicates how quickly upstream servers are able to send a response.

Note that the times shown are measured from the first request write read() to the last response byte write(); therefore, slow clients can inflate this number if TCP buffers are filled. If the median is '1000', it indicates that the median is outside the measured range, and is likely to be greater.

kbytes

This column shows the median size (in kilobytes) of successful (2xx) responses served to clients, as well as a 256k-wide histogram of responses sizes on the right.

Note that the sizes shown are actual bytes served, including headers, compression, etc. If the median is '256', it indicates that the median is at least that amount.

status codes

This column shows the distribution of status codes served to clients. They include;

0xx (grey)
No response status code sent -- Usually because the client has disconnected (i.e., aborted before any response headers have been sent). Large numbers can indicate client timeouts due to server-side delays.
1xx (white)
Informational -- rare HTTP-related messages. Mostly harmless, but should not be seen in large numbers.
2xx (green)
Success -- normal, successful responses.
3xx (blue)
Redirects, etc. -- redirections to other URLs, 300 Multiple Choices and 304 Not Modified responses. Large numbers usually indicate either redirections or 304's due to client-initiated validation (see misses).
4xx (yellow)
Client errors -- problems with the request; e.g., 401 Not Authorised, 403 Forbidden, 404 Not Found. A large number indicates that clients are making bad requests often.
5xx (red)
Server errors -- problems on the origin server and/or upstream proxies. A large number indicates that there are frequent upstream failures.
